## 封装调用
`std::unordered_map`
定义`std::unordered_map<key_type, value_type> name;`

| **操作分类**  | **常用语法**                   | **平均复杂度** | **最差复杂度** |
| --------- | -------------------------- | --------- | --------- |
| **定义**    | `unordered_map<K, V> mp;`  | $O(1)$    | $O(1)$    |
| **插入/修改** | `mp[key] = value;`         | $O(1)$    | $O(n)$    |
| **安全插入**  | `mp.insert({key, value});` | $O(1)$    | $O(n)$    |
| **快速访问**  | `mp[key]`                  | $O(1)$    | $O(n)$    |
| **检查存在**  | `mp.count(key)`            | $O(1)$    | $O(n)$    |
| **精细查找**  | `mp.find(key)`             | $O(1)$    | $O(n)$    |
| **删除元素**  | `mp.erase(key)`            | $O(1)$    | $O(n)$    |
| **删除迭代器** | `mp.erase(it)`             | $O(1)$    | $O(n)$    |
| **获取大小**  | `mp.size()`                | $O(1)$    | $O(1)$    |
| **清空**    | `mp.clear()`               | $O(n)$    | $O(n)$    |
| **容器判空**  | `mp.empty()`               | $O(1)$    | $O(1)$    |

>[!warning]
>**`operator[]` 会产生副作用**。 如果你只是写了一句 `if (count[5] > 0)`，但哈希表里原本没有 `5`，C++ 会**硬塞**一个 `{5: 0}` 进去。如果这种操作很多，哈希表会变得非常臃肿。 **安全做法**：先用 `count(num)` 检查，或者用 `find()`。

>[!tip]
>**动态扩容 (Rehash)**：当哈希表里的元素太多时（达到某个阈值，称为装载因子），它会申请更大的内存空间，把所有的 Key 重新“洒”一遍，尽量让桶变稀疏。
>**优化底层结构**：一些现代的高级实现（如 Java 8+ 的 `HashMap`）在某个桶里的冲突超过一定数量（默认 8 个）时，会把**链表**转变为**红黑树**。这样最坏情况下的复杂度会从 $O(n)$ 降低到 $O(\log n)$。
